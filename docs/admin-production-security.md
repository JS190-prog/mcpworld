# 관리자 대시보드 운영 버전 보안 설계

작성일: 2026-06-18 KST
대상: MCP World SaaS 관리자 대시보드

## 1. 목적

관리자 대시보드는 사용자 수, 결제 상태, MCPWorld 등록 상태, 프로그램별 커넥터 발급, relay 상태, 오류 이벤트를 확인하는 민감한 운영 화면이다. 운영 버전에서는 정적 프론트엔드 토큰 검사가 아니라 서버 기반 관리자 인증과 권한 검증을 적용해야 한다.

---

## 2. 운영 버전 기본 원칙

1. 관리자 화면은 프론트엔드 코드만으로 보호하지 않는다.
2. 관리자 권한은 서버에서 매 요청마다 검증한다.
3. 관리자 계정은 일반 사용자 계정과 역할을 분리한다.
4. 관리자 계정에는 추가 인증을 적용한다.
5. 관리자 작업은 감사 로그로 남긴다.
6. 통계 화면에는 민감한 원문 정보를 노출하지 않는다.
7. 파일 원문, 전체 로컬 경로, 세션 식별값 원문은 관리자 화면에도 표시하지 않는다.
8. 운영 설정값은 코드 저장소에 넣지 않는다.

---

## 3. 권장 인증 구조

운영 버전의 관리자 접근 흐름은 다음 구조를 사용한다.

```text
관리자 로그인
  -> 서버 세션 발급
  -> 관리자 역할 확인
  -> 추가 인증 확인
  -> admin API 접근 허용
  -> 모든 요청마다 권한 재검증
```

정적 목업의 토큰 입력 화면은 실제 운영에서 아래 중 하나로 대체한다.

- 관리자 전용 로그인
- 관리자 역할 기반 접근 제어
- 추가 인증 코드 확인
- 사내 IP 또는 허용 IP 제한
- 관리자 세션 만료

---

## 4. 관리자 역할 설계

운영 버전은 최소 3개 역할로 나눈다.

| 역할 | 권한 |
|---|---|
| Viewer | 통계 조회만 가능 |
| Operator | 사용자 세션 종료, relay 상태 확인, 오류 대응 가능 |
| Owner | 요금제, 관리자 계정, 보안 설정 관리 가능 |

초기 버전에서는 Viewer와 Owner만 두고, 실제 운영이 시작되면 Operator 역할을 추가해도 된다.

---

## 5. 관리자 API 설계

관리자 화면은 정적 데이터를 직접 들고 있지 않고 서버 API에서 받아온다.

권장 API:

```text
GET /api/admin/summary
GET /api/admin/programs
GET /api/admin/plans
GET /api/admin/events
GET /api/admin/relay/status
POST /api/admin/sessions/{id}/terminate
POST /api/admin/users/{id}/suspend
```

모든 admin API는 다음을 확인한다.

1. 로그인 세션 유효성
2. 관리자 역할
3. 추가 인증 통과 여부
4. 요청 IP 또는 접근 정책
5. rate limit
6. 감사 로그 기록

---

## 6. 관리자 통계에서 마스킹할 정보

관리자도 아래 정보는 원문으로 보지 않는 것을 원칙으로 한다.

| 정보 | 운영 표시 방식 |
|---|---|
| 사용자 이메일 | 일부 마스킹 또는 내부 ID 병행 |
| 로컬 파일 경로 | 프로그램명과 버전까지만 표시 |
| 세션 값 | 해시 또는 앞뒤 일부만 표시 |
| 커넥터 주소 | 전체 주소 대신 route ID 중심 표시 |
| 결제 식별자 | 결제사 고객 ID 일부 마스킹 |
| 오류 로그 | 민감 항목 제거 후 표시 |

---

## 7. 감사 로그 항목

관리자 작업은 반드시 감사 로그에 남긴다.

기록 항목:

```text
admin_user_id
role
action
resource_type
resource_id
result
ip_hash
user_agent
created_at
```

감사 로그 대상:

- 관리자 로그인
- 관리자 화면 접근
- 사용자 조회
- 세션 종료
- 사용자 정지
- 요금제 변경
- relay 재시작
- 로그 내보내기
- CSV 다운로드

---

## 8. 보안 알림 기준

다음 이벤트는 운영자에게 알림을 보내는 것이 좋다.

1. 관리자 로그인 실패 반복
2. 허용되지 않은 IP에서 관리자 접근 시도
3. 비정상적으로 많은 커넥터 발급
4. 특정 relay 오류 급증
5. 특정 사용자 세션 과다 생성
6. 프로그램별 MCP 오류율 급증
7. CSV 내보내기 과다 요청

## 10. 운영 전환 체크리스트

- [ ] 관리자 전용 로그인 또는 역할 기반 접근 제어 구현
- [ ] 관리자 추가 인증 적용
- [ ] admin API 서버 검증 적용
- [ ] 관리자 세션 만료 적용
- [ ] 관리자 감사 로그 저장
- [ ] CSV 내보내기 권한 제한
- [ ] 사용자 이메일과 세션 값 마스킹
- [ ] 로컬 파일 원문과 전체 경로 비노출
- [ ] 허용 IP 또는 접근 정책 적용
- [ ] 관리자 보안 이벤트 알림 적용
- [ ] 운영 설정값을 코드 저장소에서 분리
- [ ] 관리자 권한 변경 이력 보관

---

## 11. 현재 목업과 운영 버전 차이

| 구분 | 현재 목업 | 운영 버전 |
|---|---|---|
| 접근 제어 | 브라우저 입력값 확인 | 서버 기반 관리자 인증 |
| 통계 데이터 | 프론트엔드 목업 데이터 | admin API 응답 |
| 권한 | 단일 관리자 가정 | Viewer / Operator / Owner |
| 감사 로그 | 없음 | 모든 관리자 작업 기록 |
| 민감정보 보호 | 화면 표시 기준 | 서버 응답 단계에서 마스킹 |
| 보안 알림 | 없음 | 이상징후 알림 |

---

## 12. 결론

현재 `admin.html`의 토큰 입력은 시연용 잠금 화면이다. 운영 버전에서는 이 구조를 그대로 보안 수단으로 사용하지 말고, 서버 세션, 관리자 역할, 추가 인증, 감사 로그, API 권한 검증으로 대체해야 한다.
